- **2026-07-04 — ПЕРЕРАБОТКА механики учёбы (BS-18), отменяет механику BS-17.** Фидбек владельца: «не нравится механика и логика», «переделай полностью по топовым методичкам». Отменяет решения BS-17 «механика отбора знаю/учить/позже» и «сессия = только просроченные». Новая модель по evidence-based методикам (testing effect, learning steps, load management, active production):
  - **Один поток** вместо двух шагов: очередь занятия = просроченные + новые (лимит/день), перемешаны по темам. Отдельный экран разбора (triage) убран. Выбор владельца.
  - **Активное вспоминание обязательно**: сначала вопрос → пытаешься вспомнить → «Показать» → ответ → оценка. Нельзя пометить «знаю», не увидев ответа (главная дыра BS-17).
  - **Две стороны, слово «дорастает»**: новые/свежие — узнавание (SR→RU); зрелые (reps≥2 и интервал≥~3 дней) — говорение (RU→SR). Направление выводится из зрелости карточки, отдельного хранения на сторону нет (проще). Провал на говорении роняет слово обратно в короткие повторы. Выбор владельца: тренировать узнавание + говорение.
  - **Учебные шаги для новых/забытых**: короткие внутрисессионные показы (≈1 мин / 10 мин) перед выходом на дни. «Не помню» возвращает слово в конец текущего занятия.
  - **Дневной лимит новых** (по умолчанию ~10/день) для управления нагрузкой; счётчик newByDay в stats.
  - **«Позже» → «Убрать слово»**: bury переосмыслен как suspend (убрать из учёбы по одному, вернуть из отдельного списка), не как режим массового разбора.

- **2026-07-04 — BS-18 реализация: константы графика, состояние карточки, правило повтора в занятии.** Конкретика поверх продуктовых решений BS-18. (1) **Учебные шаги** `[1, 10]` минут (для новых и для забытых/провалившихся), интервал выпуска «на дни» на «помню» = 1 день, на «легко» = 4 дня. Ядро «дней» — прежнее SM-2-lite (ease 1.3–3.0, hard ×1.2, good ×ease, easy ×ease×1.3), «выучено» с интервала 21 день. (2) **CardProgress расширен** двумя опциональными полями: `phase: 'learning'|'review'` и `step` (индекс учебного шага). Обратная совместимость: старые записи без `phase` выводят фазу из `intervalDays` (≥1 → review). Миграция не нужна (у владельца прогресса нет). (3) **Сторона карточки не хранится**, а выводится из зрелости (`sideOf` в srs.ts): learning-фаза → узнавание; review + `reps≥2 && intervalDays≥3` → говорение; иначе узнавание. Провал (again) в review переводит слово в learning → следующая тренировка снова узнавание. (4) **Правило «вернуть в конец занятия»**: после оценки экран занятия смотрит статус нового прогресса — если он всё ещё `learning` (в т.ч. любое «Не помню», незавершённые шаги, провал review), карточка добавляется в конец очереди; graduated/review — считается завершённой. Так учебные шаги проходят внутри одного занятия без отдельного таймера. (5) **Дневной лимит новых** `NEW_PER_DAY=10` в learn.ts; счётчик `stats.newByDay: Record<день, число>` в store, инкремент при первой оценке слова (когда `progress` пуст). Очередь занятия берёт новые в пределах остатка лимита. (6) **`buried` → `suspended`** в сторе (методы `suspend`/`unsuspend`), убран `startLearning`/`makeLearning` (были нужны только triage). Экраны: удалён `triage.tsx` и компонент `SwipeCard`, `later.tsx` → `suspended.tsx` («Убранные», возврат по одному). `FlipWordCard` получил проп `side` (вопрос = русский на говорении).

Совместимость: загрузчик берёт `examples[0]` как основной пример (`exampleSr/exampleRu`), поэтому глагол корректно показывается и в карточке занятия/повторения (FlipWordCard) без спец-кода. - **2026-07-05 — ПИВОТ (платформа): Expo-приложение → устанавливаемый PWA с облачным прогрессом (BS-22).** Владелец: «хочу пользоваться с телефона всегда и не терять прогресс», нужна регистрация/вход, бэкенд — самый простой, «есть Supabase», деплой можно через GitHub Actions. Решение: **код НЕ переписываем** — тот же Expo/React Native экспортируется как web через `react-native-web` (уже в зависимостях, `web.output`). Добавляем 4 куска:
  - **Хостинг + деплой:** GitHub Pages, сборка через GitHub Actions на push в main (`expo export -p web` → `dist/` → Pages). `experiments.baseUrl = "/BytoviySerbian"` (проектная страница на сабпути), `web.output = "single"` (SPA) + `404.html = index.html` (deep-link/refresh на Pages).
  - **PWA (установка + офлайн):** `manifest.json` + service worker (`public/sw.js`, кэш оболочки), head-теги и регистрация SW через `src/app/+html.tsx`. Открывается «как приложение» с домашнего экрана; офлайн работает, т.к. контент в бандле. Онлайн нужен только для входа/синка.
  - **Бэкенд = Supabase напрямую из браузера, без своего сервера.** Auth = **email + пароль** (выбор владельца 2026-07-05; email-подтверждение выключить в дашборде для MVP). Одна таблица `profiles(id uuid pk = auth.uid(), state jsonb, updated_at timestamptz)` + RLS «строку видит только её владелец». Прогресс/сохранённое/статистика — один JSON-блоб на пользователя (состояние маленькое, по карточкам не моделируем — MVP).
  - **Синхронизация:** облако — источник правды на пользователя; локальный AsyncStorage/localStorage — офлайн-кэш и хранилище сессии. На входе тянем `state` из `profiles`, при изменениях стора — debounce-upsert (≈800 мс) в Supabase; конфликт разрешаем «последняя запись выигрывает» по `updated_at` (для одного пользователя достаточно).
  - **Секреты:** `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` через env (`.env` локально — в .gitignore; в CI — GitHub Actions secrets). anon-ключ публичный по дизайну (защита — RLS), поэтому безопасно попадает в клиентский бандл.
  Отменяет из пивота 2026-07-04 (Telegram→Expo) формулировку «без бэкенда и авторизации»: бэкенд (Supabase) и вход появляются **ради синхронизации между устройствами и несброса прогресса**. Нативная сборка (App Store/Play) не в фокусе — цель PWA; код Expo не выбрасывается, при желании нативная сборка остаётся возможной. Зависимость: `@supabase/supabase-js`.
